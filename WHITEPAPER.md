# AEON: Living Worlds — Technical Whitepaper

**Author:** Yan Desbiens — work conducted at Éthiqueia Québec inc.
**Repository:** https://github.com/Linutesto/aeon-living-worlds
**Status:** experimental prototype — broad implemented surface, active development.
**Last updated:** 2026-06

---

## Abstract

AEON is a deterministic living-world simulation coupled to a browser-based 3D dashboard, optional local LLM cognition, persistent citizens, emergent social systems, and prototype learning policies. Its central design move is clean: the simulation is the source of truth, while AI, narration, rendering, and dashboards interpret or nudge that truth rather than replacing it.

The result is closer to a civic/ecological operating system than a conventional game loop: terrain and climate create constraints; resources and cities create pressure; civilizations, citizens, religions, cultures, factions, migrations, wars, and historical sites emerge from that pressure; renderer chunks expose that state as visual evidence; optional LLM components act as chronicler, interviewer, teacher, and world-spirit.

AEON’s strongest architectural idea is its tiering:

1. Tier 0 deterministic world state — terrain, climate, resources, cities, units, events.
2. Embodied social agents — persistent citizens with traits, beliefs, memories, relations, and spatial positions.
3. Society layer — religions, factions, cultures, chronicle, interpretation.
4. Learning layer — species policies and a teacher-to-student society mind.
5. Governor layer — optional local LLM world-spirit constrained to directives.
6. Projection layer — server-side truth-to-render payloads.
7. Omega renderer — Three.js/WebGL dashboard with chunks, LOD, instancing, overlays, texture packs, day/night, and performance controls.

The current codebase already implements a large amount of this vision. The main risks are scale pressure, save compatibility, mixed prototype maturity across AI systems, and a need for stronger dirty-propagation/data-oriented boundaries as simulation richness grows.

## 1. Project Identity

AEON describes itself as a browser-based living world simulation where civilizations rise, fracture, migrate, starve, worship, fight, remember, and evolve. The README frames it as an experimental sandbox rather than a finished game. That framing is accurate: the project has real architecture, tests, APIs, renderer systems, saves, and cognition prototypes, but balance, AI behavior, and formats remain in active flux.

The repository is a Python backend plus browser frontend:

- Backend package: aeon/
- Web dashboard: web/
- Config: config.yaml (plus optional local overrides)
- Docs: docs/, plus README.md, PERF.md, RENDERER.md, PLAN.md, ROADMAP.md
- Saves: saves/aeon_saves.sqlite, saves/policy_weights/
- Tests: tests/

The package metadata declares the project as aeon-living-worlds with Python >=3.11 and dependencies on FastAPI, Uvicorn, NumPy, PyYAML, HTTPX, and Pydantic. Runtime entry point is aeon.__main__:main (run with `python -m aeon`).

## 2. Architectural Thesis

AEON’s core thesis is: a believable world should be generated by pressure, memory, and consequence, not by decorative random events.

Several explicit rules appear across docs and implementation:

- Same seed + same config should produce the same world.
- Renderer records must derive from simulation state.
- Roads, cities, landmarks, citizens, scars, lights, and lore should not be invented client-side.
- LLMs may interpret, narrate, teach, or issue bounded directives, but the deterministic simulation owns outcomes.
- Scale is managed through chunking, LOD, materialized citizens, batching, and optional AI tiers.

That philosophy is unusually coherent. The code follows it in several places:

- aeon/sim/world.py builds deterministic world state and advances it through ordered subsystem steps.
- aeon/sim/worldgen.py centralizes restartable generation variables and clamps unsafe inputs.
- aeon/render/projection.py generates Omega payloads from existing simulation state only.
- aeon/governor/directives.py limits world-spirit effects to bounded operations.
- aeon/governor/arbiter.py schedules scarce LLM calls rather than letting ambient narration starve critical cognition.
- aeon/persistence.py saves full opaque world state and stores policy weights separately.

## 4. Runtime Flow

The core deterministic loop is in aeon/sim/world.py:

```text
world.tick += 1
terrain.step(world)
climate.step(world)
resources.step(world)
species.step(world)
evolution.step(world)
civilization.step(world)
cities.step(world)
roads.step(world)
units.step(world)
events.step(world)
remember_historical_sites(world, new_events)
age transient markers
```



This order matters:

- Terrain and climate mutate environmental constraints first.
- Resources update before species and cities consume pressure signals.
- Species and evolution run before civilization/city updates.
- Cities update before roads and units, allowing settlement/economy changes to affect route and movement behavior.
- Events decay last, and notable events are converted into persistent historical sites.

The design is easy to reason about because each tick has a single pipeline. The downside is that, as systems grow, this ordered monolith can become a bottleneck unless hot paths move toward cached aggregates and dirty propagation.

## 5. Determinism and World Generation

WorldGenConfig is one of the strongest pieces of the project.

It separates fields into three classes:

1. Structural fields: seed, width, height, starting species, starting population, starting civilizations.
2. Simulation params: clamped knobs from sim/params.BOUNDS.
3. Presentation fields: graphics preset, texture pack, LOD distance, render budgets, densities.

This is clean because player-editable generation variables and governor-editable simulation parameters share the same bounded registry. The same sea_level, war_propensity, or resource knob means the same thing whether it comes from UI, API, config, restart, or governor directive.

World generation uses _build_valid_world_shell, which attempts up to 24 candidate seeds derived from the base seed. It scores terrain by landmass fraction, buildable tiles, river tiles, and target land ratio. This is a good compromise: deterministic, but not blindly accepting broken topologies.

Generation flow:

```text
WorldState(seed)
terrain.generate
climate.initialize
resources.seed
terrain.analyze_terrain
terrain.validate_terrain
species.seed
civilization.seed_initial
terrain.compute_buildable_score
roads.rebuild
```



Notable design win: genesis starts with multiple distinct civilizations by default, not a single homogeneous people. This gives the simulation immediate political texture.

## 6. World State Model

WorldState stores dense numeric grids, collections of live entities, event markers, historical memory, and monotonic ID counters.

Major fields include:

- Terrain/climate/resource arrays.
- Buildability, roads, and road graph.
- Species, civilizations, cities, units.
- Transient markers.
- Persistent historical sites.
- Active world events.
- Genesis events.
- ID counters.

The design favors practical Python dataclasses and NumPy arrays. This is good for prototyping and testability. The tradeoff is that long-term scale may require a more data-oriented layout for hot loops, especially if world size and population budgets increase.

## 7. Civilizations

Civilizations are seeded at genesis through aeon/sim/civilization.py. The current test suite verifies:

- The world starts with five distinct civilizations by default.
- Each civ has full identity.
- Founding events are emitted.
- Capitals are spaced apart.
- Citizens inherit national character.
- Collapse, merge, split, and golden age behavior exists.
- Civ identity serializes for renderer payloads.

The docs and PLAN indicate civs carry:

- People identity.
- Color.
- Ideology.
- Ideology axes.
- Cultural traits.
- Preferred desires.
- Economic, military, religious, and exploration bias.
- Diplomatic stance.
- Capital city.
- Parent civ and merge/collapse status.
- Golden-age metadata.

This is exactly the right shape for an emergent sim because it supports both macro politics and micro citizen inheritance.

The main future risk is making diplomacy and memory deeper without turning civ lifecycle into random event soup. The best next step is to make diplomatic memory and grievance explicit: wars, betrayals, shared religions, trade dependence, famine aid, old ruins, and migration waves should all affect future civ decisions.

## 8. Cities and Economy

Cities are the middle layer between macro civilization state and individual citizens. They aggregate population, economy, buildings, resources, demographics, unrest, migration pressure, knowledge, faith, class, and visible renderer signals.

The current architecture uses cities as compression nodes:

- Thousands or millions of imagined inhabitants can be represented by aggregate city stats.
- Only a bounded persona pool is materialized as Person objects.
- Renderer payloads derive district, skyline, building, lights, and overlays from city state.

That is the right move. AEON should not simulate every villager equally at all times. Cities are the pressure vessels; citizens are sampled embodied representatives.

The strongest renderer-facing idea is that skyline, district identity, night lights, scars, and overlays should communicate real simulation pressure: wealth, trauma, education, unrest, plague, famine, religion, knowledge, militarization.

---

## 9. Citizens and Spatial Embodiment

Citizens live in aeon/agents/. The central objects are:

- Person
- Relationship
- MemoryStore
- PopulationManager
- spatial helpers in spatial.py
- trait/action utilities in traits.py

Population has a global persona budget of 4000. This suggests a deliberate level-of-detail design: keep enough persistent individuals to create continuity, but avoid full-population object explosion.

Citizens include personality, skills, goals, beliefs, profession, class, relationships, memories, and now spatial position/movement state. The spatial module contains pathfinding, passability, route steering, target choice, observations, migration opportunity, danger zones, terrain risk, nearest resources, social targets, and feature-vector generation.

This is a major upgrade path for AEON. Spatial embodiment turns citizens from abstract dice rolls into entities that can plausibly:

- Go home.
- Go to work.
- Flee.
- Migrate.
- Seek food.
- Visit markets.
- Join factions.
- React to nearby danger.
- Produce observations for learning systems.

The risk is CPU cost. Pathfinding and target selection can become expensive if many citizens update often. The code already has active embodied citizen caps in config, which is wise.

## 10. Society Layer

The society layer includes:

- beliefs.py
- religion.py
- faction.py
- culture.py
- chronicle.py
- interpret.py

This layer converts raw survival/economic/political pressure into meaning.

Religions can found, spread, adopt followers, schism, and trigger holy wars. Factions can found, recruit, exert influence, and revolt. Cultures can found, spread, and modify city identity. Beliefs derive ideology and grievances from person/city/world conditions.

This is where AEON becomes interesting. Terrain and cities create the physics; society creates the story. The Chronicle and interpretation layer then turn factual events into human-readable history, biographies, city descriptions, newspapers, religion summaries, culture summaries, and discovery reports.

The strongest constraint is that interpretation should never invent factual state. It can style and compress, but facts must come from world/city/person/society state.

## 11. Historical Memory

remember_historical_sites persists material world memory from notable events. This is a crucial system.

Event types map to site kinds:

- war/battle/holy_war/revolution → battlefield
- collapse → ruin
- famine → famine marker
- plague → plague marker
- settlement → foundation
- religion founded → shrine
- schism → schism site
- culture → monument
- economy → market crisis
- migration → migration waypoint
- discovery → discovery site

The docs emphasize that these scars are render evidence only and never invented. This is excellent design. Historical sites make the world legible and persistent: old disasters can become terrain scars, ruins, shrines, abandoned farms, monuments, and discovery markers.

The next architectural leap is to make history feed back into the simulation more strongly:

- Trauma should alter migration and settlement appeal.
- Heritage should strengthen culture and identity.
- Battlefields should affect diplomacy and religion.
- Old roads and ruins should influence future trade and settlement.
- Repeated famine zones should become known risk regions.

AEON already has early heritage/trauma signals. This should become one of the project’s signature mechanics.

## 12. AI Layer: Species Policies

aeon/ai/species_policy.py implements per-species learning policies. It supports a Torch path and NumPy fallback. The model uses action features from citizen/spatial observations and learns over a bounded replay buffer.

Current training design uses:

- Advantage-Weighted Regression.
- Entropy bonus/floor.
- KL trust region.
- Bounded replay buffer.
- Lower learning rates than earlier REINFORCE-style training.

This is a good correction. The PLAN explicitly notes that off-policy REINFORCE over a large stale replay buffer caused behavior drift. AWR with KL and entropy is more stable for this setting.

The key design question is what these species policies are supposed to become:

- Are they creature/ecology policies?
- Citizen action priors?
- Civ-level instincts?
- A small learning substrate for local adaptation?

The current architecture can support all of these, but the recommendation is to keep their mandate narrow: species policies should learn bounded behavioral tendencies from survival outcomes, not become general intelligence. General cognition belongs to the society mind and LLM teacher tiers.

## 13. AI Layer: Society Mind

The society mind in aeon/mind/ is one of AEON’s most ambitious systems.

Core components:

- cohort.py: groups citizens into teacher cohorts.
- teacher.py: calls an LLM teacher and parses JSON decisions.
- dataset.py: stores supervised samples.
- encode.py: encodes actions, emotions, intents, targets, embeddings.
- liquid.py: defines a CfC-style liquid neural network and double-buffering.
- trainer.py: trains the student network.
- curriculum.py: controls teacher/student handoff phases.
- runtime.py: orchestrates HybridMind decisions and status.

The intended flow:

```text
citizens/world state
 |
 v
CohortBatcher groups routine/crisis citizens
 |
 v
LLM teacher labels actions/emotions/intents/targets
 |
 v
SocietyDataset stores samples
 |
 v
SocietyTrainer trains LiquidSocietyNet
 |
 v
DoubleBufferedNet publishes weights safely
 |
 v
HybridMind lets student drive bounded population share
 |
 v
TeacherCurriculum gates autonomy by metrics/drift
```



This is conceptually strong because it uses the expensive LLM as a teacher, not as the runtime brain for everyone. The student network then amortizes learned behavior across many citizens.

The curriculum is especially important. Tests verify:

- Teacher-first warmup.
- No premature takeover with weak metrics.
- Independent metric gates.
- Drift blocks promotion.
- Severe regression rolls back.
- Status serialization.

This is the right safety shape for a self-training sim. The biggest technical risk is label quality and distribution shift. If the teacher’s outputs are inconsistent or too stylized, the student learns mush. The best mitigation is richer evaluation: agreement by action, emotion, intent, target, city crisis type, profession, class, religion, and faction membership.

## 14. LLM Governor and Scheduler

The governor stack is optional and local-first, via Ollama. It includes:

- llm.py: HTTP client.
- prompts.py: world-spirit prompt and parser.
- directives.py: bounded directive application.
- memory.py: myths and decisions.
- governor.py: orchestration.
- arbiter.py: LLM scheduler.

The scheduler is a standout subsystem. It recognizes that one local GPU cannot serve every LLM consumer concurrently without thrashing VRAM.

Consumers are prioritized:

- cohort_teacher: 0
- citizen_interview: 1
- spirit_governor: 2
- rare_citizen: 3
- major_event: 4
- world_report: 5
- chronicle: 6
- news: 7
- flavor: 8
- narration: 8
- diagnostics: 9

Protected consumers are teacher, interview, and governor. Lower-priority work can age upward but cannot enter the protected band. The scheduler implements cooldowns, quotas, rolling token budget, deduplication, stale cancellation, fallback values, and history logging.

This is exactly what a local-LLM-heavy sim needs. Without it, ambient flavor murders the useful cognition. With it, AEON can safely run a 27B teacher, a small spirit model, interviews, chronicle generation, and flavor without turning the GPU into soup.

## 15. Rendering Architecture

The renderer architecture is built around server-side projection and client-side chunk streaming.

aeon/render/projection.py produces:

- Manifest payload.
- Chunk payloads.
- Entity payloads.
- Policy inspector payloads.
- Counterfactual payloads.

Chunks contain:

- Terrain fields.
- Terrain features.
- Rivers.
- Roads.
- Bridges.
- Shorelines.
- Districts.
- Buildings.
- Skylines.
- Citizens.
- Units.
- Scars.
- Overlays.

This is a good API boundary: the client asks for chunks around the camera, and the server sends compact simulation-derived records. The renderer should not invent truth. It only visualizes truth.

The renderer docs describe the desired information hierarchy:

1. Terrain communicates biome, elevation, slope, fertility, water, snow, wetland, farms, scars, ruins, old roads.
2. Cities communicate wealth, poverty, population density, education, faith, militarization, industry, stability, famine, plague, war, unrest.
3. Districts communicate identity, density, prosperity, damage, material, activity.
4. Roads and routes communicate trade, migration, military purpose.
5. Night lighting communicates wealth, density, knowledge, faith, unrest, plague/famine, culture.
6. History markers age and fade visually but persist as world memory.

The current roadmap mentions IBL, derived normal maps, bloom, texture packs, quality presets, instancing, LOD, performance HUD, and mobile discipline. This is a serious renderer direction, not just debug triangles.

## 16. Placement and Believability

aeon/render/placement.py handles deterministic collision-free city layout. It uses:

- Spatial hash rejection.
- Footprint radii.
- Road clearance.
- Slope limits.
- Density falloff.
- Deterministic golden-angle placement.
- Memoized city-local computation.

Tests verify spacing, density falloff, slope rejection, stats structure, and sane constants.

This system matters more than it sounds. Believability often dies in the last 10%: overlapping buildings, roads through houses, cities on cliffs, white terrain flashes. AEON’s placement layer is an example of simulation-first visual polish.

## 18. Persistence

aeon/persistence.py stores save slots in SQLite. The save table contains:

- Slot.
- Timestamp.
- Tick.
- World name.
- Seed.
- Manual flag.
- Summary JSON.
- Weights path.
- Opaque state blob.

World state is pickled. Policy weights are stored separately as native torch files.

This is pragmatic and fast for a prototype. The tradeoff is long-term compatibility. Pickled full-state saves are fragile across refactors. The project already acknowledges save compatibility may change.

Recommendation: keep pickle for dev autosaves, but eventually add a versioned semantic save format for stable releases:

```text
save.json/zstd
 version
 config
 params
 world arrays
 entity tables
 event history
 society state
 governor memory
 mind metadata
weights/*.pt
```



Until then, the current system is fine for rapid iteration.

## 20. Current Strengths

### 20.1 Coherent Source-of-Truth Philosophy

The biggest win: AEON consistently separates truth from presentation. Renderer, LLM, chronicle, and UI are downstream of deterministic state.

### 20.2 Deterministic Restartable Worldgen

WorldGenConfig is a strong foundation for experimentation. Same seed + same config matters enormously for debugging.

### 20.3 Multi-Tier Cognition

LLM teacher → dataset → liquid student → curriculum-gated autonomy is ambitious and structurally sound.

### 20.4 Local-LLM Resource Scheduling

The scheduler is exactly the kind of unsexy but essential infrastructure that makes agentic systems usable on one GPU.

### 20.5 Simulation-Derived Renderer

The renderer architecture can become visually beautiful without lying. That is rare and powerful.

### 20.6 Historical Sites as Material Memory

Persistent scars are the seed of AEON’s identity. The world remembers in geography, not only text.

---

## 21. Main Weaknesses and Risks

### 21.1 Python Object Scale

Dataclasses and dictionaries are perfect for prototyping, but hot-loop performance may degrade as the world becomes richer. City-level aggregates help, but profiling will eventually force data-oriented caches.

### 21.2 Save Compatibility

Pickle saves are fragile. Fine now, risky later.

### 21.3 AI Maturity Variance

Some AI systems are strong architecture but prototype implementation. The teacher/student stack needs robust evaluation, label QA, and long-run drift control.

### 21.4 Renderer Payload Growth

Omega chunks are rich. Without dirty propagation and payload budgeting, API bandwidth and JSON encoding can become the bottleneck.

### 21.5 Event Semantics

As more events are added, AEON must avoid random event spam. Every event should have causes, consequences, visible evidence, and memory.

### 21.6 LLM Prompt Fragility

Governor, chronicle, interpretation, and teacher systems depend on parseable LLM output. The code includes fallbacks and parsers, but more schema validation and adversarial tests will help.

## 22. Recommended Next Architecture Passes

### Pass A — Dirty Propagation and Payload Budgets

Add version counters to world/city/district/history/render-relevant state.

```text
city.version.economy
city.version.demographics
city.version.buildings
city.version.society
world.version.terrain
world.version.roads
world.version.historical_sites

```


Then chunk payloads can skip unchanged sections or send compact diffs.

### Pass B — Long-Run Soak Harness

Create a script that runs 10k-100k ticks headless and records:

- tick ms p50/p95/p99
- memory usage
- city/civ/person counts
- event rates
- scheduler history
- mind drift
- save/load success
- payload size samples
- determinism hash every N ticks

### Pass C — Historical Feedback 2.0

Turn historical sites into a first-class influence field:

- trauma map
- heritage map
- sacred map
- ruined infrastructure map
- old battlefield diplomacy modifiers
- famine memory settlement penalties

### Pass D — Data-Oriented City Cache

Keep dataclasses as semantic API, but add arrays/caches for hot city computations:

```text
city_ids[]
population[]
food[]
wealth[]
unrest[]
faith[]
education[]
position_y[]
position_x[]
```



### Pass E — Mind Evaluation Dashboard

Expose the teacher/student system with real diagnostics:

- action confusion matrix
- agreement by cohort type
- drift by class/profession/city crisis
- student confidence distribution
- teacher call cost
- fallback rate
- replay sample age

### Pass F — Semantic Save Format

Keep pickle dev saves, add versioned export saves.

## 25. Conclusion

AEON is already more than a toy simulation. Its architecture has a real spine: deterministic world state, pressure-driven systems, persistent agents, social emergence, optional bounded LLM cognition, and a renderer that visualizes truth instead of inventing beauty.

The project’s best idea is not “LLM NPCs” or “procedural terrain.” It is the combination of:

```text
deterministic simulation
+ persistent memory
+ bounded local AI
+ material historical evidence
+ readable 3D visualization
```



That combination can produce worlds that feel authored without being authored.

The next leap is not adding more random features. It is strengthening causality: every famine, war, migration, religion, ruin, road, city light, faction, and myth should be traceable back to pressure in the world and should leave consequences future systems can feel.

If AEON keeps that rule, it has a real shot at becoming a strange, rich, locally-run living-world engine — the kind of thing you open for five minutes and accidentally watch until 3 a.m. because some tiny mountain republic just survived a famine, built a shrine on the old battlefield, and taught its children to hate the sea.

Absolute goblin-grade simulation energy. In the best way.
